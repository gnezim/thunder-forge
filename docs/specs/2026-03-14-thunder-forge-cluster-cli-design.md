# Thunder Forge Cluster CLI — Design Spec

## 1. Overview

Transform the `thunder-forge` repository from a Telegram bot into a cluster management CLI for a local MLX inference cluster: 4× Mac Studio M4 Max (128 GB each) + 1× Radxa ROCK 5 ITX+ (32 GB, ARM64 Linux).

The CLI manages the full lifecycle: config generation, model distribution, service deployment, and health checks. All state lives in a single YAML file (`node-assignments.yaml`) — no database, no lock files.

**Repository:** `https://github.com/shared-goals/thunder-forge`

### 1.1 What's in scope

- Clean the existing repo (delete all code, keep `README.md` and `LICENSE`)
- `configs/node-assignments.yaml` — single source of truth for models, nodes, and assignments
- `docker/docker-compose.yml` — LiteLLM + Open WebUI + PostgreSQL (3 services)
- CLI commands: `generate-config`, `ensure-models`, `deploy`, `health`
- `scripts/setup-node.sh` — first-time node bootstrap
- `.github/workflows/deploy.yml` — GitOps auto-deploy

### 1.2 What's out of scope

- **Monitoring stack (phase 2):** Grafana, VictoriaMetrics, VictoriaLogs, Vector, macOS exporter, `promscrape.yml`, `grafana.ini`, `grafana-datasources.yaml`
- **Backup:** `thunder-forge backup`, `scripts/backup.sh`, systemd timer
- **Telegram bot:** deleted, rebuild from scratch if needed later

### 1.3 Network

| Node | IP | Role |
|------|----|------|
| rock | 192.168.1.61 | Infrastructure (Docker, LiteLLM, Open WebUI) |
| msm1 | 192.168.1.101 | Inference (vllm-mlx) |

> **Note:** SSH user on all nodes: `admin`.

## 3. `node-assignments.yaml`

Single source of truth for the entire cluster. All CLI commands read this file.

```yaml
# ── Model registry ──────────────────────────────────
models:
  coder:
    source:
      type: huggingface
      repo: "mlx-community/Qwen3-Coder-Next-4bit"
      revision: "main"
    disk_gb: 44.8
    kv_per_32k_gb: 8
    active_params: "3B of 80B"
    max_context: 131072
    notes: "70%+ SWE-Bench ≈ Sonnet"

# ── Nodes ──────────────────────────────────────────
nodes:
  rock: { ip: "192.168.1.61", ram_gb: 32, user: "admin", role: infra }
  msm1: { ip: "192.168.1.101", ram_gb: 128, user: "admin", role: inference }
  msm2: { ip: "192.168.1.102", ram_gb: 128, user: "admin", role: inference }
  msm3: { ip: "192.168.1.103", ram_gb: 128, user: "admin", role: inference }
  msm4: { ip: "192.168.1.104", ram_gb: 128, user: "admin", role: inference }

# ── Node assignments ────────────────────────────────
assignments:
  msm1:
    - model: coder
      port: 8000
  msm2:
    - model: coder
      port: 8000
  msm3:
    - model: coder
      port: 8000
  msm4:
    - model: coder
      port: 8000
```

### 3.1 Model registry fields

| Field | Required | Description |
|-------|----------|-------------|
| `source.type` | yes | `huggingface`, `convert`, `local`, `pip` |
| `source.repo` | yes (hf/convert) | HuggingFace repo ID |
| `source.revision` | no | Git ref (default: `main`) |
| `source.quantize` | yes (convert) | Quantization bits (e.g. `"4"`) |
| `source.path` | yes (local) | Absolute path on target node |
| `source.package` | yes (pip) | pip package name |
| `source.weight_repo` | no (pip) | HF repo for pre-downloading weights |
| `disk_gb` | yes | Model weight size on disk |
| `kv_per_32k_gb` | yes (llm) | KV cache at 32K context |
| `ram_gb` | no | Runtime RAM override (for pip models) |
| `active_params` | no | Human-readable active parameter count |
| `max_context` | yes (llm) | Maximum context length |
| `serving` | no | `embedding`, `mlx-openai-server`, `cli` (default: vllm-mlx) |
| `notes` | no | Human-readable notes |

### 3.2 Assignment slot fields

| Field | Required | Description |
|-------|----------|-------------|
| `model` | yes | Key from `models` registry |
| `port` | yes (llm) | Port for vllm-mlx |
| `embedding` | no | If `true`, adds `--embedding-model` flag |

### 4.3 `generate-config`

```
thunder-forge generate-config [--check]
```

**Reads:** `configs/node-assignments.yaml`
**Writes:** `configs/litellm-config.yaml`

**Steps:**

1. Parse `node-assignments.yaml`
2. Validate memory budgets per node:
   - For each node: `sum(model.disk_gb + model.kv_per_32k_gb for each slot) + 8 (OS) <= node.ram_gb`
   - If a model has `ram_gb` set, use `ram_gb` instead of `disk_gb` (pip-installed tools)
   - Print per-node breakdown: `msm1: coder(44.8+8kv) + 8 OS = 60.8 GB / 128 GB ✅`
   - Exit with error on any violation
3. Generate `litellm-config.yaml`:
   - For each `{node, port, model}` with no `serving` key or `serving: vllm-mlx`:
     - Emit `hosted_vllm/<repo>` entry with `api_base: http://<node_ip>:<port>/v1`
     - Set `max_input_tokens` from `max_context`, `max_output_tokens: 16384`
   - For `serving: embedding` — skip (loaded via `--embedding-model` flag on the vllm-mlx process)
   - For `serving: mlx-openai-server` — emit `openai/<repo>` entry
   - For `serving: cli` — skip (no API server)
   - When a slot has `embedding: true`, emit additional entry for the embedding model on the same `api_base`
   - Each model entry must include `api_key: "none"` (vllm-mlx backends have no auth)
   - Settings: `routing_strategy: least-busy`, `num_retries: 2`, `timeout: 120`, `allowed_fails: 3`, `cooldown_time: 30`, `callbacks: ["prometheus"]`, `health_check_interval: 30`
   - Master key: `os.environ/LITELLM_MASTER_KEY`
4. Write header comment: `# AUTO-GENERATED by thunder-forge generate-config`

> **Important:** `api_base` must end with `/v1` (e.g. `http://192.168.1.101:8000/v1`).
> Without it, LiteLLM hits `/chat/completions` instead of `/v1/chat/completions`
> and vllm-mlx returns 404.

> **Phase 2 note:** `promscrape.yml` generation is deferred to the monitoring phase.
> The `callbacks: ["prometheus"]` setting is included now (zero-cost) so LiteLLM
> exposes `/metrics` when monitoring is added later.

**`--check` mode:**
- Generate in memory, compare with committed file
- Exit 0 if identical, exit 1 with diff if mismatch
- Used in CI to enforce config/assignments sync

### 4.4 `ensure-models`

```
thunder-forge ensure-models [--dry-run]
```

**Reads:** `configs/node-assignments.yaml`

**Steps:**

1. Parse assignments, collect unique `{model, node}` pairs
2. For each model, by source type:

| Source | Action |
|--------|--------|
| `huggingface` | `huggingface-cli download <repo> --revision <rev>` on rock → `rsync` from HF cache to each assigned node |
| `convert` | Download on rock → SSH to an inference node → `mlx_lm.convert` → rsync converted weights to all assigned nodes |
| `local` | SSH to node, verify path exists |
| `pip` | SSH to node, `uv tool install <package>`; if `weight_repo`, also `huggingface-cli download <weight_repo>` |

3. Idempotent — skip already-cached models
4. `--dry-run` — print what would be downloaded/synced without doing it

**Paths:**

| Source | Download location (rock) | Sync target (inference node) | Idempotency check |
|--------|--------------------------|------------------------------|--------------------|
| `huggingface` | `~/.cache/huggingface/hub/models--<org>--<name>/` | `~/.cache/huggingface/hub/models--<org>--<name>/` | Snapshot dir exists on target |
| `convert` | `~/.cache/huggingface/` (source) | `~/.cache/mlx-models/<model-key>/` | Converted dir exists on target |
| `local` | N/A | Path from `source.path` | Path exists on target |
| `pip` | N/A | Installed via `uv tool` | `uv tool list` shows package |

**Error handling:** If a download fails, abort before syncing. Per-node sync errors are reported but don't stop other nodes.

### 4.5 `deploy`

```
thunder-forge deploy [--node NODE]
```

**Orchestration sequence:**

1. **Ensure models** — run `ensure-models` logic
2. **Generate config** — validate memory, write `litellm-config.yaml`
3. **Deploy launchd plists** — for each `{node, port, model}`:
   - Generate plist from template with:
     - `Label: com.vllm-mlx-<port>`
     - `ProgramArguments`: vllm-mlx serve with model repo, `--port`, `--host 0.0.0.0`, `--continuous-batching`, `--max-model-len <max_context>`
     - Optional: `--embedding-model <repo>` if slot has `embedding: true`
     - `EnvironmentVariables`: PATH, HOME
     - `StandardOutPath/StandardErrorPath`: `~/logs/vllm-mlx-<port>.log/.err`
     - `RunAtLoad: true`, `KeepAlive: true`, `ThrottleInterval: 10`, `ProcessType: Interactive`
   - SCP plist to `~/Library/LaunchAgents/com.vllm-mlx-<port>.plist` on node
   - Obtain UID on remote node: `ssh admin@<ip> id -u`
   - `launchctl bootout gui/<uid>/com.vllm-mlx-<port>` (ignore error if not loaded)
   - `launchctl bootstrap gui/<uid> ~/Library/LaunchAgents/com.vllm-mlx-<port>.plist`
4. **Deploy newsyslog config** — SCP log rotation config to `/etc/newsyslog.d/vllm-mlx.conf` on each node:
   ```
   # logfilename                             [owner:group] mode count size(KB) when  flags
   /Users/admin/logs/vllm-mlx-*.log            admin:staff     644  7     102400   *     CNJ
   /Users/admin/logs/vllm-mlx-*.err            admin:staff     644  7     102400   *     CNJ
   ```
   Phase 1 includes only vllm-mlx entries. Vector/macOS exporter entries added in phase 2.
   Flags: `C` = copytruncate, `N` = no signal, `J` = bzip2. See `guides/inference-nodes.md` §3.3.
5. **Clean stale plists** — list `com.vllm-mlx-*.plist` on each node, remove any with ports not in current assignments
6. **Restart LiteLLM** — on rock: `cd ~/thunder-forge/docker && docker compose restart litellm`
7. **Health poll** — for each `{node, port}`:
   - Poll `http://<ip>:<port>/v1/models` every 5s, up to 180s
   - Success = HTTP 200 (model loaded and serving)
   - Timeout = warning, continue to next

**`--node NODE`:** Restrict steps 1–5 to a single node. Steps 6–7 always run.
Exit with error if NODE is not found in assignments.

> **Plist template reference:** See `guides/inference-nodes.md` §3.2 for the full
> reference plist structure. The CLI generates plists using `xml.etree.ElementTree`
> (no external dependencies).

### 4.6 `health`

```
thunder-forge health
```

**Checks:**

1. **Inference nodes** — for each `{node, port}` in assignments:
   - HTTP GET `http://<ip>:<port>/v1/models` — ✅ if 200, ❌ if error/timeout
2. **Infrastructure (rock)** — SSH to rock, `docker compose ps --format json`:
   - Check each service status (litellm, openwebui, postgres)

**Output:**

```
=== Inference ===
  msm1:8000 (coder): ✅
  msm2:8000 (coder): ✅
  msm3:8000 (coder): ✅
  msm4:8000 (coder): ✅

=== Infrastructure ===
  LiteLLM:    ✅
  Open WebUI: ✅
  PostgreSQL: ✅

=== Model Assignments ===
  msm1: coder:8000
  msm2: coder:8000
  msm3: coder:8000
  msm4: coder:8000
```

Exit code: 0 if all healthy, 1 if any check fails.

## 5. Docker Compose

**`docker/docker-compose.yml`** — three services, no monitoring:

```yaml
name: thunder-forge

services:
  postgres:
    image: postgres:16-alpine
    container_name: postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: litellm
      POSTGRES_USER: litellm
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-litellm-local}
    volumes:
      - postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U litellm"]
      interval: 10s
      timeout: 3s
      retries: 3
    networks: [infra]

  litellm:
    image: ghcr.io/berriai/litellm:main-stable
    container_name: litellm
    restart: unless-stopped
    ports:
      - "4000:4000"
    volumes:
      - ../configs/litellm-config.yaml:/app/config.yaml:ro
    command:
      - "--config"
      - "/app/config.yaml"
      - "--port"
      - "4000"
      - "--host"
      - "0.0.0.0"
      - "--num_workers"
      - "4"
    environment:
      LITELLM_MASTER_KEY: ${LITELLM_MASTER_KEY:-sk-local-cluster-key}
      DATABASE_URL: postgresql://litellm:${POSTGRES_PASSWORD:-litellm-local}@postgres:5432/litellm
      UI_USERNAME: ${UI_USERNAME:-admin}
      UI_PASSWORD: ${UI_PASSWORD:-changeme}
    depends_on:
      postgres: { condition: service_healthy }
    healthcheck:
      test: ["CMD", "python", "-c",
        "import urllib.request; urllib.request.urlopen('http://localhost:4000/health/readiness')"]
      interval: 15s
      timeout: 5s
      retries: 3
    networks: [infra]

  openwebui:
    image: ghcr.io/open-webui/open-webui:latest
    container_name: openwebui
    restart: unless-stopped
    ports:
      - "8080:8080"
    volumes:
      - openwebui-data:/app/backend/data
    environment:
      OPENAI_API_BASE_URLS: http://litellm:4000/v1
      OPENAI_API_KEYS: ${LITELLM_MASTER_KEY:-sk-local-cluster-key}
      ENABLE_OLLAMA_API: "false"
      WEBUI_AUTH: ${WEBUI_AUTH:-true}
      WEBUI_SECRET_KEY: ${WEBUI_SECRET_KEY:-change-this-secret}
      WEBUI_NAME: LLM Cluster
      ENABLE_SIGNUP: ${ENABLE_SIGNUP:-true}
    depends_on:
      litellm: { condition: service_healthy }
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:8080/health"]
      interval: 15s
      timeout: 5s
      retries: 3
    networks: [infra]

networks:
  infra:
    driver: bridge

volumes:
  postgres-data:
  openwebui-data:
```

**`docker/.env.example`:**

```bash
LITELLM_MASTER_KEY=sk-local-cluster-key
POSTGRES_PASSWORD=litellm-local
UI_USERNAME=admin
UI_PASSWORD=changeme
WEBUI_SECRET_KEY=change-this-secret
WEBUI_AUTH=true
ENABLE_SIGNUP=true
```

### 5.1 Design decisions (learnings from MVP testing)

- **Healthcheck uses Python, not curl** — LiteLLM image doesn't ship curl
- **Healthcheck endpoint is `/health/readiness`** — `/health` requires API key
- **PostgreSQL required** — LiteLLM admin UI returns "Not connected to DB" without it
- **Open WebUI: `latest` tag** — `main` can serve stale cached versions
- **`WEBUI_AUTH=true` by default** — unlike MVP where we used `false` for testing

## 6. `setup-node.sh`

Single bootstrap script with two modes.

```
curl -fsSL https://raw.githubusercontent.com/shared-goals/thunder-forge/main/scripts/setup-node.sh | bash -s -- inference
curl -fsSL https://raw.githubusercontent.com/shared-goals/thunder-forge/main/scripts/setup-node.sh | bash -s -- infra
```

### 6.1 Inference role (macOS)

1. Install Homebrew (if not present)
2. Install uv (official installer)
3. Install vllm-mlx (`uv tool install vllm-mlx`)
4. Disable macOS sleep (`sudo pmset -a sleep 0 displaysleep 0 disksleep 0`)
5. Create `~/logs` directory
6. Print summary of what was installed

**Does NOT install (phase 2):** Vector, macOS exporter.

### 6.2 Infra role (Linux ARM64)

1. Install Docker Engine (official convenience script)
2. Add user to docker group
3. Install uv (official installer)
4. Clone thunder-forge repo to `~/thunder-forge`
5. `cd ~/thunder-forge && uv sync`
6. Generate `docker/.env` with random secrets (`openssl rand -hex 16`)
7. `cd ~/thunder-forge/docker && docker compose up -d`
8. Generate SSH key (`ssh-keygen -t ed25519`)
9. Print summary + reminder to distribute SSH keys

### 6.3 What stays manual

- GitHub Actions runner setup (needs interactive token from GitHub UI)
- SSH key distribution to inference nodes
- Model downloads (`thunder-forge ensure-models` after all nodes are bootstrapped)

## 7. GitHub Actions

**`.github/workflows/deploy.yml`:**

```yaml
name: Deploy Cluster

on:
  push:
    branches: [main]
    paths:
      - 'configs/**'
      - 'src/thunder_forge/**'
      - 'docker/**'
  workflow_dispatch:

concurrency:
  group: deploy-cluster
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: [self-hosted, Linux, ARM64, infra]
    timeout-minutes: 30

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install dependencies
        run: uv sync

      - name: Validate configs
        run: |
          uv run thunder-forge generate-config --check
          cd docker && docker compose config --quiet

      - name: Ensure models
        run: uv run thunder-forge ensure-models
        timeout-minutes: 120

      - name: Deploy
        run: uv run thunder-forge deploy

      - name: Health check
        run: |
          sleep 10
          uv run thunder-forge health

      - name: Report
        if: always()
        run: |
          echo "## Deploy Summary" >> $GITHUB_STEP_SUMMARY
          echo "- **Trigger:** ${{ github.event_name }}" >> $GITHUB_STEP_SUMMARY
          echo "- **Commit:** ${{ github.sha }}" >> $GITHUB_STEP_SUMMARY
          echo "" >> $GITHUB_STEP_SUMMARY
          echo '```' >> $GITHUB_STEP_SUMMARY
          uv run thunder-forge health 2>&1 >> $GITHUB_STEP_SUMMARY || true
          echo '```' >> $GITHUB_STEP_SUMMARY
```

## 8. Implementation order

Config-first approach — each step produces a testable result.

### Step 1: Repo cleanup and skeleton

- Delete all existing code (keep `README.md`, `LICENSE`, `.git/`)
- Create directory structure
- Write `pyproject.toml`, `.gitignore`
- Stub `cli.py` with Typer app and placeholder commands
- Write `configs/node-assignments.yaml` with coder model on all 4 nodes
- Write `docker/docker-compose.yml` and `docker/.env.example`

**Testable:** `uv sync` succeeds, `uv run thunder-forge --help` shows commands.

### Step 2: `generate-config`

- Implement `cluster/config.py`:
  - YAML parser for node-assignments
  - Memory validator
  - LiteLLM config generator
  - `--check` mode
- Write tests for memory validation and config generation

**Testable:** `uv run thunder-forge generate-config` produces valid `litellm-config.yaml`. Run `docker compose up` on rock with the generated config → LiteLLM starts.

### Step 3: `health`

- Implement `cluster/health.py`:
  - HTTP checks for vllm-mlx endpoints
  - Docker service checks on rock via SSH

**Testable:** `uv run thunder-forge health` reports status of all services.

### Step 4: `ensure-models`

- Implement `cluster/models.py`:
  - HuggingFace download on rock
  - rsync to inference nodes
  - `--dry-run` mode

**Testable:** `uv run thunder-forge ensure-models` downloads coder model, syncs to msm1.

### Step 5: `deploy`

- Implement `cluster/deploy.py`:
  - Plist template generation
  - SSH deploy to inference nodes
  - launchctl bootstrap/bootout
  - newsyslog config deployment
  - Stale plist cleanup
  - LiteLLM restart
  - Health poll

**Testable:** `uv run thunder-forge deploy --node msm1` deploys to one node. Full deploy after.

### Step 6: `setup-node.sh` and GitHub Actions

- Write `scripts/setup-node.sh` (inference + infra modes)
- Write `.github/workflows/deploy.yml`

**Testable:** Bootstrap a fresh node with the script. Push a config change → GitHub Actions deploys automatically.

## 9. Key design decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| SSH | `subprocess.run(["ssh", ...])` | No Fabric/Paramiko. SSH keys are a prerequisite. Simple, debuggable. |
| State | Stateless (no DB, no locks) | All state in `node-assignments.yaml` + what's running on nodes. |
| Repo root detection | `git rev-parse --show-toplevel` or walk parents for `configs/node-assignments.yaml` | Works from any subdirectory. |
| Error handling | Per-node errors don't stop other nodes | Report all failures at the end with ✅/❌ summary. |
| Python version | 3.12+ | Latest stable, required by latest dependency versions. |
| Package manager | uv only | Fast, handles venvs transparently, used for both dev and deployment. |
| Config generation | No Jinja2 — pure Python string building | One less dependency. Templates are simple enough. |
| Plist generation | `xml.etree.ElementTree` | Structured XML, no external dep. Reference: `guides/inference-nodes.md` §3.2 |
| SSH user | `user` field in `nodes` config (default: `admin`) | Explicit per-node, avoids hardcoding assumptions |

## 10. Client configuration

After deployment, clients connect to the cluster via:

```bash
# OpenAI SDK v1.0+ compatible tools:
export OPENAI_BASE_URL=http://192.168.1.61:4000/v1
export OPENAI_API_KEY=sk-local-cluster-key

# Browser chat:
# http://192.168.1.61:8080
